# Mawari Web (PoC)

「あと◯分使えるなら、ここを歩こう」を一発で提案する Mawari の Web 版プロトタイプ。
出発地・目的地・**許容できる追加時間**を入れると、最短経路と比べた**回り道候補**を
「所要時間差」「運賃差額」「歩行距離・歩数・カロリー(推定)」付きで提示します。

## 構成

- **バックエンドなしの静的サイト**（`index.html` / `styles.css` / `app.js`）。
- 経路データは [Transit API (ls8h)](https://api.transit.ls8h.com/) をブラウザから直接取得
  （無認証・読み取り専用・CORS有効: `Access-Control-Allow-Origin: *`）。
- ビルド不要。そのままホスティング（GitHub Pages / Netlify / 任意の静的配信）に置けます。

## ローカルで動かす

ブラウザのCORS/モジュール制約を避けるため、簡易サーバ経由で開くのを推奨します。

```bash
cd web
python3 -m http.server 8000
# → http://localhost:8000/ を開く
```

## ロジック概要（`app.js`）

1. **最短経路**を `/api/v1/plan`（`numItineraries=1`）で取得し、基準の所要時間 `T0`・運賃 `F0` とする。
2. **回り道候補**を収集:
   - 複数経路 `/api/v1/plan?numItineraries=5`
   - 鉄道回避 `/api/v1/plan?avoidModes=rail,subway`（バス＋徒歩寄りの候補）
   - **末尾区間の徒歩置換**: 基準経路の末尾の鉄道区間（1〜2区間）を降りて、
     そこから目的地まで `avoidModes=rail,subway,bus,…` の**徒歩のみ検索**でつなぐ
     合成経路（例: 押上で降りて曳舟まで歩く）。
3. 経路の「形」で重複排除（発車時刻違いは最速のみ残す）。
4. **許容追加時間内**（`deltaSecs ≤ budget`）の候補を、運賃が安い順 → 追加時間が短い順で並べる。
   基準の運賃が不明な場合は、**運賃が判明している候補を先頭**に出す
   （徒歩置換で運賃データのある区間まで乗車した候補など）。
5. 各候補に **所要時間差・運賃差額・徒歩量（時間/距離/歩数/カロリー）** を付与して表示。

## 既知の制約（重要）

- **運賃はフィード依存**: 東京メトロ・都営・路線バス等は運賃が出るが、
  **JR各線・新幹線・私鉄の一部は運賃データが無く「運賃データなし」**になる。
  経路に運賃不明の区間が1つでも含まれると、その経路全体の運賃が出ない
  （詳細は `../docs/api-evaluation-transit-ls8h.md`）。
  → 対策として、**運賃不明の末尾区間を徒歩に置き換えた候補**を併せて提示する。
  運賃データのある区間まで乗車して残りを歩くため、その候補では運賃が表示でき、
  かつ運賃不明だった区間ぶんを実質的に節約できる。
- **徒歩距離・歩数・カロリーは推定値**（API が徒歩距離[m]を返さないため、徒歩時間 × 約80m/分で換算）。
- **「手前下車」候補の精度**: ls8h には「指定地点付近の駅を列挙する」手段が無く、
  経路レッグも途中停車駅を返さないため、現状の回り道生成は「複数経路＋鉄道回避」ベース。
  より精緻な手前下車/先歩きは自前バックエンド（`../docs/design-self-hosted-backend.md`）で対応予定。

## テスト

`app.js` の探索ロジックは実APIに対する統合テストと、`page.route` でレスポンスを
モックしたブラウザUIスモークテストで動作確認済み（出発/目的地のサジェスト→選択→検索→
最短経路＋回り道カード描画、コンソールエラーなし）。
